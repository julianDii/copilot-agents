## Single Source of Truth

All instructions live in **one file**:

```text
instructions/global-copilot-instructions.md
```

Edit that file when you want to change your instructions. Then run the sync script to deploy the changes. **Never edit `.github/copilot-instructions.md` or `~/.config/...` directly** — they are generated outputs.

---

## How Instructions Get to Copilot

```text
instructions/global-copilot-instructions.md   ← EDIT THIS
        │
        ├─ ./scripts/sync-agents.sh --global
        │         │
        │         ├──► ~/.config/github-copilot/intellij/global-copilot-instructions.md
        │         │    (JetBrains — active in ALL projects on this machine, globally)
        │         │
        │         └──► .github/copilot-instructions.md
        │              (per-repo fallback — active when this repo is open in any IDE)
        │
        └─ .vscode/settings.json points here
             (VS Code — active in this workspace automatically)
```

## Setup on a New Machine

### JetBrains IDEs (IntelliJ IDEA, PyCharm, WebStorm, etc.)

```bash
git clone <this-repo>
cd copilot-agents
chmod +x scripts/sync-agents.sh
./scripts/sync-agents.sh --global
```

JetBrains Copilot will now apply the instructions to **every project** you open — no per-project configuration needed.

> **If you already have a global instructions file**, the script will back it up as
> `global-copilot-instructions.md.backup-<timestamp>` before overwriting.
> Merge any custom content into `instructions/global-copilot-instructions.md` and re-run.

Restart your JetBrains IDE after running the script.

### VS Code

**Option 1: Workspace-scoped (this repo only)**

The `.vscode/settings.json` in this repo already points to `instructions/global-copilot-instructions.md`, so instructions are active automatically when you open this repo. No setup needed.

**Option 2: Global (all VS Code projects)**

To apply these instructions to every project you open in VS Code:

1. Go to **Settings → search `codeGeneration.instructions`**
2. Add: `{ "file": "<absolute-path-to-this-repo>/instructions/global-copilot-instructions.md" }`
3. Replace `<absolute-path-to-this-repo>` with the actual path (e.g., `/Users/yourname/repos/copilot-agents`)

> **Note**: Unlike JetBrains which has a single global instructions file location, VS Code requires you to specify the absolute path in your user settings.

## Updating the Instructions

1. Edit `instructions/global-copilot-instructions.md`
2. Run `./scripts/sync-agents.sh --global`
3. Restart your JetBrains IDE (IntelliJ IDEA, PyCharm, etc.)

VS Code picks up changes automatically — no restart needed.

## Deploying to Another Project

To give a specific project its own copy of the instructions — useful for teammates who don't have the global setup, or for CI/repo-level enforcement:

```bash
./scripts/sync-agents.sh ~/IdeaProjects/my-other-project
```

Installs into the target project:

- `.github/copilot-instructions.md` — picked up automatically by Copilot in any IDE when that repo is open
- `.github/agents/<name>.agent.md` — JetBrains agent (picked up automatically)
- `.github/agents/<name>.vscode.agent.md` — VS Code agent (generated automatically with correct tool names)

> `evals/` is **not** copied — it is a QA tool for this repo only. Fixtures and snapshots are used to validate the agents here; target projects don't need them.
> **If the target project already has a `.github/copilot-instructions.md`**, it will be backed up with a timestamp before being replaced.

### How agents work in each IDE

**JetBrains (IntelliJ IDEA, PyCharm, WebStorm, etc.)**

Agents in `.github/agents/*.agent.md` are **automatically discovered** when you open this repo — no sync script needed. They appear in the Copilot Chat mode/agent dropdown.

**VS Code**

Agents in `.github/agents/` are discovered **only if they use VS Code tool names** (`codebase`, `problems`, `terminal`). The source agents in this repo use JetBrains tool names, so:

- **In this repo (without sync)**: Agents won't work in VS Code because tool names don't match
- **After running sync-agents.sh to deploy to a project**: The script generates `*.vscode.agent.md` variants with correct tool names automatically

Type `@` in Copilot Chat to select an agent from the dropdown (if available).

> **TL;DR**: JetBrains works out-of-the-box with this repo. VS Code needs the sync script to generate compatible agent variants.

### Providing context for review

The agents have codebase tools enabled — they **can** search and read files in the open workspace autonomously. However, giving explicit context still produces faster, more focused reviews.

#### How context works

| Scenario | What happens |
|----------|-------------|
| You attach a file | Agent reviews exactly that — no drift into unrelated files |
| You paste a diff | Agent reviews only the changed lines |
| You describe a function by name | Agent searches the codebase to find and read it |
| No context given | Agent searches the codebase but may cast too wide a net |

**Best practice: attach the primary file + let the agent chase dependencies.** For example, attach a `views.py` file and the agent will use the codebase tool to look up related serializers, models, and permission classes on its own.

#### What to attach or paste

| What you want reviewed | What to provide |
|------------------------|----------------|
| A single file | Open the file → attach it with `#` or the paperclip (📎) |
| A specific function or block | Select the code in the editor → paste it directly |
| A pull request / diff | Paste the `git diff` output or the PR diff from GitHub |
| Multiple related files | Attach each file individually with `#` |
| An error or failing test | Paste the full stack trace + attach the relevant file |

#### How to get a git diff to paste

```bash
# Diff of your branch vs main
git diff main...HEAD

# Diff of the last commit only
git diff HEAD~1

# Diff of staged changes (before committing)
git diff --cached
```

#### Tips for better reviews

- **Be specific about scope**: *"Review only the authentication logic in this file"* gets a tighter review than *"review this file"*.
- **State the stack**: if it's not obvious from the code, mention it — *"Django 4.2, DRF, PostgreSQL"*.
- **Flag what you're unsure about**: *"I'm not confident about the transaction boundary on line 45"* directs the agent where it matters most.
- **Use prototype mode for drafts**: add *"this is a prototype, not production code"* to skip production-grade checks you don't need yet.

## Tools Reference

> ⚠️ **CRITICAL: Tool names are IDE-specific.** JetBrains (IntelliJ IDEA, PyCharm, WebStorm, etc.) and VS Code use completely different tool names for the same capabilities. This is why agents written for one IDE won't work in the other without conversion.
>
> **The source agents in this repo use JetBrains tool names** because this is a JetBrains-first repo. When deploying to a project with `sync-agents.sh`, the script **automatically generates VS Code variants** of every agent — you only maintain one source file.

### How the dual-variant approach works

```text
.github/agents/code-review.agent.md          ← source (JetBrains tool names)
        │
        └─ sync-agents.sh <target>
                  │
                  ├──► <target>/.github/agents/code-review.agent.md          (JetBrains)
                  └──► <target>/.github/agents/code-review.vscode.agent.md   (VS Code)
```

The VS Code variant is identical to the JetBrains one — only the `tools:` line is rewritten automatically.

### JetBrains tool names

```yaml
tools: ['read_file', 'semantic_search', 'grep_search', 'file_search', 'list_dir', 'get_errors']
```

| Tool | What it does | Typical use case |
|------|-------------|-----------------|
| `read_file` | Read the full contents of a file by path | Follow an import, inspect a config, read a related model or serializer |
| `semantic_search` | Find relevant code by concept/meaning across the codebase | Understand existing patterns, locate similar implementations without knowing the exact file |
| `grep_search` | Search for an exact string or regex across all files | Find all usages of a symbol, audit a setting, check for a pattern (e.g. `@csrf_exempt`) |
| `file_search` | Find files by name or glob pattern | Locate a file without knowing its full path (e.g. find all `serializers.py`, locate a test file) |
| `list_dir` | List the contents of a directory | Understand project structure, find which modules exist, navigate an unfamiliar repo layout |
| `get_errors` | Read current compiler, linter, or type-checker errors from the IDE | Reason about existing errors before suggesting fixes, validate a fix resolves an issue |
| `run_in_terminal` | Execute a shell command | Run tests, build commands, or scripts — use only in agents that need to act, not just review |
| `get_terminal_output` | Read output from a previously run terminal command | Check the result of a command run by the agent |

### VS Code tool names

VS Code Copilot uses a different, shorter set of tool names. The sync script maps between them automatically:

| VS Code tool | What it does | JetBrains equivalent |
|-------------|-------------|----------------------|
| `codebase` | Search and read files in the workspace | `read_file` + `semantic_search` + `grep_search` + `file_search` + `list_dir` |
| `problems` | Read linter/compiler errors | `get_errors` |
| `terminal` | Run shell commands | `run_in_terminal` + `get_terminal_output` |
| `fetch` | Fetch content from a URL | *(no JetBrains equivalent)* |
| `githubRepo` | Query GitHub API (issues, PRs, commits) | *(no JetBrains equivalent)* |

### Choosing tools for your agent

Write agents using JetBrains tool names — the sync script handles VS Code conversion. Use `tools: []` only if you need the agent to work in both IDEs without running the sync script first.

| Pattern | JetBrains `tools:` | Use when |
|---------|-------------------|---------|
| Prompt-only | `[]` | Agent only needs what the user pastes — works in both IDEs without syncing |
| Read-only codebase | `['read_file', 'semantic_search', 'grep_search', 'file_search', 'list_dir']` | Review and analysis — agent can follow imports and find patterns |
| Full diagnostic | `['read_file', 'semantic_search', 'grep_search', 'file_search', 'list_dir', 'get_errors']` | Debugging — agent needs live IDE error output |
| Autonomous | `['read_file', 'semantic_search', 'grep_search', 'file_search', 'list_dir', 'get_errors', 'run_in_terminal', 'get_terminal_output']` | Agent needs to execute commands — use with care |

## Testing Agents & Instructions

Use the fixtures and rubric in **[evals/EVALS.md](evals/EVALS.md)** to verify agents are working correctly and to catch regressions after any change.

**Quick start:**

1. Select an agent from the Copilot Chat dropdown
2. Attach a fixture file from `evals/fixtures/` with `#`
3. Type `Review this file.`
4. Score the output against the [rubric](evals/EVALS.md#rubric-score-each-criterion-0-or-1) — 7/8 is a pass
5. Save passing outputs to `evals/snapshots/` and `git diff` after future changes

Each fixture is designed to contain exactly one or two known issues at a known severity — making it easy to spot if the agent misses a finding, inflates severity, or fabricates issues that aren't there.

## Contributing

1. **Change instructions:** edit `instructions/global-copilot-instructions.md`, run `./scripts/sync-agents.sh --global`.
2. **Add an agent:** add `<name>.agent.md` to `.github/agents/` following the frontmatter format (`description` + `tools: [...]`), refer to the Tools Reference above for valid values, then update the Agents table above.
3. **Add prompts:** add examples to `PROMPTS.md` under the relevant section.
4. **Add fixtures:** for any new agent or checklist item, add a fixture to `evals/fixtures/` and a test case row to `evals/EVALS.md`.
5. Bump `VERSION` and add an entry to `CHANGELOG.md`.
