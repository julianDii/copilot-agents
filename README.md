## Setup on a New Machine

### JetBrains

```bash
git clone <this-repo>
cd copilot-agents
chmod +x scripts/sync-agents.sh
./scripts/sync-agents.sh --global
```

JetBrains Copilot will now apply the instructions to **every project** you open тАФ no per-project configuration needed.

> **If you already have a global instructions file**, the script will back it up as
> `global-copilot-instructions.md.backup-<timestamp>` before overwriting.
> Merge any custom content into `instructions/global-copilot-instructions.md` and re-run.

Restart your JetBrains IDE after running the script.

### VS Code

VS Code instructions are workspace-scoped. The `.vscode/settings.json` in this repo already points to `instructions/global-copilot-instructions.md`, so instructions are active automatically when you open this repo.

For global VS Code instructions (all projects), go to:
**Settings тЖТ search `codeGeneration.instructions` тЖТ add** `{ "file": "<absolute-path>/instructions/global-copilot-instructions.md" }`.

## Deploying to Another Project

To give a specific project its own copy of the instructions тАФ useful for teammates who don't have the global setup, or for CI/repo-level enforcement:

```bash
./scripts/sync-agents.sh ~/IdeaProjects/my-other-project
```

Installs into the target project:

- `.github/copilot-instructions.md` тАФ picked up automatically by Copilot in any IDE when that repo is open
- `.github/agents/<name>.agent.md` тАФ JetBrains agent (picked up automatically)
- `.github/agents/<name>.vscode.agent.md` тАФ VS Code agent (generated automatically with correct tool names)

> `evals/` is **not** copied тАФ it is a QA tool for this repo only. Fixtures and snapshots are used to validate the agents here; target projects don't need them.
> **If the target project already has a `.github/copilot-instructions.md`**, it will be backed up with a timestamp before being replaced.

## What the Instructions Cover

The 9 sections in `global-copilot-instructions.md` shape Copilot's behaviour across the full SDLC:

| # | Section | Purpose |
|---|---------|---------|
| 1 | Response Format & Behaviour | Code-first output, no filler, self-verification, language matching |
| 2 | Hard Constraints тАФ Never Do | Explicit guardrails: no silent file creation, no undeclared deps, no unsafe execution, no broken interfaces |
| 3 | Engineering Baseline | Evidence-based, production quality, minimal safe changes |
| 4 | Requirements & Design | Assumptions list + milestones + risks + acceptance criteria |
| 5 | Implementation | Patterns, readability, input validation, deps/versioning, Django/React specifics, a11y |
| 6 | Testing | Happy path + boundary + failure, auth tests, deterministic, fixtures |
| 7 | Code Review Mindset | Failure modes first, ORM issues, severity ordering, тЪа BREAKING flag |
| 8 | Debugging & Incident Response | Ranked hypothesis list, observability, blast-radius |
| 9 | Documentation, Commits & PR Hygiene | Conventional Commits, branch naming, PR description format |

### Providing context for review

The agents have codebase tools enabled тАФ they **can** search and read files in the open workspace autonomously. However, giving explicit context still produces faster, more focused reviews.

#### How context works

| Scenario | What happens |
|----------|-------------|
| You attach a file | Agent reviews exactly that тАФ no drift into unrelated files |
| You paste a diff | Agent reviews only the changed lines |
| You describe a function by name | Agent searches the codebase to find and read it |
| No context given | Agent searches the codebase but may cast too wide a net |

**Best practice: attach the primary file + let the agent chase dependencies.** For example, attach a `views.py` file and the agent will use the codebase tool to look up related serializers, models, and permission classes on its own.

#### What to attach or paste

| What you want reviewed | What to provide |
|------------------------|----------------|
| A single file | Open the file тЖТ attach it with `#` or the paperclip (ЁЯУО) |
| A specific function or block | Select the code in the editor тЖТ paste it directly |
| A pull request / diff | Paste the `git diff` output or the PR diff from GitHub |
| Multiple related files | Attach each file individually with `#` |
| An error or failing test | Paste the full stack trace + attach the relevant file |

#### How to get a git diff to paste

```bash
# Diff of your branch vs main
git diff main...HEAD

# Diff of the last commit only
git diff HEAD~1

# Diff of staged changes (before committing)
git diff --cached
```

#### Tips for better reviews

- **Be specific about scope**: *"Review only the authentication logic in this file"* gets a tighter review than *"review this file"*.
- **State the stack**: if it's not obvious from the code, mention it тАФ *"Django 4.2, DRF, PostgreSQL"*.
- **Flag what you're unsure about**: *"I'm not confident about the transaction boundary on line 45"* directs the agent where it matters most.
- **Use prototype mode for drafts**: add *"this is a prototype, not production code"* to skip production-grade checks you don't need yet.

## Tools Reference

> тЪая╕П **Tool names are IDE-specific.** JetBrains and VS Code use different names for the same capabilities. The source agents in this repo use JetBrains tool names. When deploying to a project with `sync-agents.sh`, the script **automatically generates a VS Code variant** of every agent alongside the JetBrains one тАФ you only maintain one source file.

### How the dual-variant approach works

```text
.github/agents/code-review.agent.md          тЖР source (JetBrains tool names)
        тФВ
        тФФтФА sync-agents.sh <target>
                  тФВ
                  тФЬтФАтФАтЦ║ <target>/.github/agents/code-review.agent.md          (JetBrains)
                  тФФтФАтФАтЦ║ <target>/.github/agents/code-review.vscode.agent.md   (VS Code)
```

The VS Code variant is identical to the JetBrains one тАФ only the `tools:` line is rewritten automatically.

### JetBrains tool names

```yaml
tools: ['read_file', 'semantic_search', 'grep_search', 'file_search', 'list_dir', 'get_errors']
```

| Tool | What it does | Typical use case |
|------|-------------|-----------------|
| `read_file` | Read the full contents of a file by path | Follow an import, inspect a config, read a related model or serializer |
| `semantic_search` | Find relevant code by concept/meaning across the codebase | Understand existing patterns, locate similar implementations without knowing the exact file |
| `grep_search` | Search for an exact string or regex across all files | Find all usages of a symbol, audit a setting, check for a pattern (e.g. `@csrf_exempt`) |
| `file_search` | Find files by name or glob pattern | Locate a file without knowing its full path (e.g. find all `serializers.py`, locate a test file) |
| `list_dir` | List the contents of a directory | Understand project structure, find which modules exist, navigate an unfamiliar repo layout |
| `get_errors` | Read current compiler, linter, or type-checker errors from the IDE | Reason about existing errors before suggesting fixes, validate a fix resolves an issue |
| `run_in_terminal` | Execute a shell command | Run tests, build commands, or scripts тАФ use only in agents that need to act, not just review |
| `get_terminal_output` | Read output from a previously run terminal command | Check the result of a command run by the agent |

### VS Code tool names

VS Code Copilot uses a different, shorter set of tool names. The sync script maps between them automatically:

| VS Code tool | What it does | JetBrains equivalent |
|-------------|-------------|----------------------|
| `codebase` | Search and read files in the workspace | `read_file` + `semantic_search` + `grep_search` + `file_search` + `list_dir` |
| `problems` | Read linter/compiler errors | `get_errors` |
| `terminal` | Run shell commands | `run_in_terminal` + `get_terminal_output` |
| `fetch` | Fetch content from a URL | *(no JetBrains equivalent)* |
| `githubRepo` | Query GitHub API (issues, PRs, commits) | *(no JetBrains equivalent)* |

### Choosing tools for your agent

Write agents using JetBrains tool names тАФ the sync script handles VS Code conversion. Use `tools: []` only if you need the agent to work in both IDEs without running the sync script first.

| Pattern | JetBrains `tools:` | Use when |
|---------|-------------------|---------|
| Prompt-only | `[]` | Agent only needs what the user pastes тАФ works in both IDEs without syncing |
| Read-only codebase | `['read_file', 'semantic_search', 'grep_search', 'file_search', 'list_dir']` | Review and analysis тАФ agent can follow imports and find patterns |
| Full diagnostic | `['read_file', 'semantic_search', 'grep_search', 'file_search', 'list_dir', 'get_errors']` | Debugging тАФ agent needs live IDE error output |
| Autonomous | `['read_file', 'semantic_search', 'grep_search', 'file_search', 'list_dir', 'get_errors', 'run_in_terminal', 'get_terminal_output']` | Agent needs to execute commands тАФ use with care |

## Testing Agents & Instructions

Use the fixtures and rubric in **[evals/EVALS.md](evals/EVALS.md)** to verify agents are working correctly and to catch regressions after any change.

**Quick start:**

1. Select an agent from the Copilot Chat dropdown
2. Attach a fixture file from `evals/fixtures/` with `#`
3. Type `Review this file.`
4. Score the output against the [rubric](evals/EVALS.md#rubric-score-each-criterion-0-or-1) тАФ 7/8 is a pass
5. Save passing outputs to `evals/snapshots/` and `git diff` after future changes

Each fixture is designed to contain exactly one or two known issues at a known severity тАФ making it easy to spot if the agent misses a finding, inflates severity, or fabricates issues that aren't there.
